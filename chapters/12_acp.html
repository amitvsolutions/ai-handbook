    Chapter 12 · Agents Talking to Agents

    Agent Communication Protocols: the emerging peer layer

    MCP let each of Exponential AI's agents reach tools and data. But an agent is not a tool — it plans, holds state, and can say "no." As agents multiply, they need to talk to each other as peers. Agent Communication Protocols (ACP) are the emerging category aiming to standardise that. This chapter covers the intent honestly: what problem ACP targets, IBM's ACP as a notable example, and why the wire formats are still converging.

    ~17 min read Level 1 → Level 3 Mermaid + ASCII diagrams Quiz included
    01 · Recap & Introduction

    Where Exponential AI left off

    Chapter 11 solved integration sprawl: every tool and data source now exposes one MCP server, and every agent reaches it through one client. But re-read that sentence — MCP connects a model to tools. The relationship is asymmetric: a caller invokes a passive capability that always answers. Exponential AI's returns agent calling the fraud agent is a different shape entirely. Both are autonomous. Either can be busy, can ask a clarifying question, can decline, can take minutes to finish. Treating another agent as if it were a stateless tool loses everything that makes it an agent.

    So a new question emerges: how do independent agents — possibly built by different teams, frameworks, or companies — discover each other, describe their capabilities, and exchange messages as peers? That is the domain of Agent Communication Protocols (ACP). Be clear up front: ACP is best understood today as an emerging category and intent, not a single settled wire format. Several efforts are converging on it, and this chapter describes the shape of the problem and the leading examples rather than declaring a winner.

    Emerging-terminology note

    "ACP" is used by more than one initiative and the specifics are still moving. We describe the category accurately, name IBM's Agent Communication Protocol as a notable reference, and flag in NOTES.md that wire formats are still converging. Don't treat any single detail here as a ratified standard.

    02 · Why this technology exists

    Why the world is reaching for ACP

    ACP exists because the "agent as tool" abstraction breaks down at scale. When Exponential AI had one agent, it just called functions. When it had a handful inside one process, LangGraph wired them together (Chapter 10). But agents are increasingly built by different owners: the fraud agent belongs to the risk team, the logistics agent to operations, and tomorrow a supplier's agent belongs to another company entirely. You cannot put all of them in one graph. They need a shared way to talk across process, team, and organisational boundaries.

    The requirements that fall out of that are distinctly agent-shaped: discovery (find an agent and learn what it can do), capability description (a machine-readable profile), asynchronous tasks (requests that take time and stream progress), stateful conversations (multi-turn, with identity and context), and negotiation (an agent may accept, decline, or ask for more). A tool-calling protocol like MCP was never meant to carry those semantics. ACP is the category of standards trying to.

    03 · Problem statement

    The problem, stated precisely

    Given many autonomous agents owned by different parties, define a common protocol for discovering agents, advertising their capabilities, and exchanging messages/tasks as peers — with support for asynchronous, long-running, multi-turn interactions and clear identity — so agents built independently can collaborate without bespoke, pairwise glue.

    The honest difficulty is that this is being solved right now, in public, by several groups at once. IBM's ACP frames it one way; Google's A2A (next chapter) frames it another; other proposals exist. They agree strongly on the goals and differ on the wire details. The mature architectural stance is to design to the category — discovery, capability profiles, async tasks, identity — while keeping the specific protocol behind an adapter so you can adopt whichever format wins.

    Three levels of understanding

    One idea, three depths

    Level 1 · Simple

    A shared language for coworkers

    MCP is like a worker using tools. ACP is like two workers talking to each other — introducing themselves, asking for help, and replying — using an agreed language.

    Level 2 · Engineering

    Peer messaging + discovery

    ACP-style protocols add agent discovery, a capability profile, and asynchronous, multi-turn message/task exchange between independent agents — semantics tool-calling lacks.

    Level 3 · Architect

    An interop category, not one spec

    Design to the category (discovery, capabilities, async tasks, identity) behind an adapter. Adopt a concrete spec (IBM ACP, Google A2A) without hard-coupling your system to it.

    04 · Historical evolution

    From RPC to agent peers

    The idea of machines talking to machines is old; what's new is talking to reasoning, stateful peers.

    Classic RPC / SOA

    Services call each other — but callees are stateless functions, not autonomous decision-makers.

    Agent frameworks (2023–24)

    Multi-agent systems appear, but agents talk only within one framework/process.

    MCP (late 2024)

    Standardises agent→tool access — deliberately not agent↔agent.

    ACP efforts (2025)

    IBM's ACP, Google's A2A and others push to standardise peer agent communication; wire formats still converging.

    05 · Internal architecture

    What an ACP-style protocol contains

    Across the leading efforts, the building blocks rhyme even where the encodings differ. First, identity & discovery: an agent has an addressable endpoint and a way to be found. Second, a capability profile — a machine-readable description of what the agent does, its inputs/outputs, and how to reach it (Google's A2A calls this an "Agent Card"; IBM's ACP has its own descriptor). Third, a message/task model: a request creates a task that may run asynchronously, stream intermediate updates, and end in a result — because agents don't answer instantly.

    Fourth, conversation state: interactions are multi-turn, so the protocol carries a task/session identity that lets both sides keep context across messages. Fifth, trust & auth: since a peer may belong to another team or company, authentication, authorization, and (increasingly) verifiable identity matter more than in the tool case. Note what's shared with MCP — schemas, transports like JSON/HTTP — and what's genuinely new: symmetry. In ACP either side can initiate, decline, or negotiate. IBM positions its ACP around RESTful, standards-friendly agent messaging; the details are still settling, which is exactly why we design to these blocks rather than to one byte layout.

    It's worth dwelling on why symmetry changes the engineering so much. In the tool model, failure is simple: the tool errors and the caller retries or gives up. Between peers, failure is a conversation — the callee might be temporarily overloaded and ask you to retry later, might partially complete and hand back what it has, or might refuse on policy grounds it will not disclose. That means an ACP-style protocol has to carry status as a first-class concept (accepted, working, input-needed, failed, completed) and let both sides reason about it. It also means idempotency and correlation IDs move from "nice to have" to "mandatory": if Exponential AI's returns agent re-sends a request after a timeout, the fraud agent must recognise it as the same task rather than scoring the order twice. These are the unglamorous details that separate a real peer protocol from a pretty diagram, and they are precisely the details the competing efforts are still refining.

    06 · Step-by-step execution

    Returns agent consults the fraud agent

    1. Discover. The returns agent looks up the fraud agent and fetches its capability profile.
    2. Address. It sees a "risk_assessment" capability with an input/output contract.
    3. Open a task. It sends a message creating an async task with the order details.
    4. Stream. The fraud agent acknowledges, works, and streams progress ("checking device history…").
    5. Negotiate. It asks a clarifying question ("include chargeback history?"); the returns agent replies.
    6. Result. The fraud agent returns a risk score; the task closes with an auditable trail.

    A peer task exchange, sketched (illustrative, not a ratified format)

    // returns-agent → fraud-agent : create task
{"type":"task.create","capability":"risk_assessment",
 "input":{"order_id":90214,"amount":260.00},
 "reply_to":"agent://returns","task_id":"t-77a1"}

// fraud-agent → returns-agent : streamed update, then result
{"type":"task.update","task_id":"t-77a1","status":"working"}
{"type":"task.result","task_id":"t-77a1",
 "output":{"risk":"low","score":0.12}}
    07 · Analogies

    Four ways to picture ACP

    Two professionals emailing across companies: they introduce themselves, state what they need, and go back and forth — versus one person pressing a vending-machine button (that's MCP).

    A B2B contract interface: Exponential AI's agent and a supplier's agent agree on a common way to request and fulfil work across the org boundary.

    Like the early days of web services before REST settled: everyone agreed on the need (services calling services) while SOAP/REST/other formats competed.

    Telephone standards before one prevailed: the goal (any phone calls any phone) was clear long before a single signalling standard won.

    11 & 12 · Advantages & disadvantages

    Strengths and costs

    Advantages

    • Cross-boundary collaboration between independently built agents.
    • Agent-shaped semantics: async, multi-turn, negotiable.
    • Discovery of agents you didn't pre-wire.
    • Ecosystem potential — an internet of agents.

    Disadvantages

    • Still converging — no single ratified wire format yet.
    • Fragmentation risk — multiple competing efforts.
    • Trust & security are hard across orgs.
    • Overhead vs a simple in-process call when peers are co-located.
    13 & 14 · Best practices & anti-patterns

    Adopting ACP wisely

    Best practices

    • Design to the category; keep the concrete spec behind an adapter.
    • Model long work as async tasks, not blocking calls.
    • Authenticate peers and log every cross-agent exchange.
    • Version capability profiles explicitly.

    Anti-patterns

    • Hard-coupling your whole system to one draft spec.
    • Treating peers as stateless tools.
    • Trusting unauthenticated agents across boundaries.
    • Synchronous blocking on long agent tasks.

    16 · Architect notes

    Think like an architect

    Trade-off analysis

    The bet is timing. Adopt the category now to be ready for peer collaboration, but don't hard-wire to a draft. An adapter around discovery + capability profile + async tasks lets Exponential AI ride whichever spec (IBM ACP, Google A2A) prevails without rewrites.

    Failure scenario

    Exponential AI hard-codes one early draft everywhere. Six months later the ecosystem consolidates on a different format, and every agent integration must be rewritten. The lesson: interoperability layers should be swappable while a standard is still converging.

    17 · Common mistakes

    Where teams go wrong

    • Assuming ACP is one finished standard.
    • Wrapping peer agents as dumb tools, losing autonomy.
    • Ignoring cross-org identity and trust.
    • Blocking synchronously on long-running agent tasks.

    19 · Checklist

    Before wiring agents as peers

    • The callee really is an autonomous agent, not a tool.
    • Concrete protocol sits behind a swappable adapter.
    • Long work modelled as async tasks with streaming.
    • Peers are authenticated; exchanges are audited.
    • Capability profiles are versioned.
    • NOTES.md flag reviewed — no single standard assumed.

    21 · Summary

    Summary

    Once agents outnumber the graphs that can hold them and start crossing team and company lines, they need to talk as peers — not as tools. Agent Communication Protocols are the emerging category aiming to standardise that: discovery, capability profiles, asynchronous multi-turn tasks, and cross-boundary identity. We named IBM's ACP as a notable reference and were deliberately honest that the wire formats are still converging; the architect's move is to design to the category behind a swappable adapter rather than betting the enterprise on one draft.

    That honesty raises a fair question: if the category is still settling, is there a concrete, documented specification an architect can actually read and build against today? There is — and it's the most fully realised of the peer protocols.
