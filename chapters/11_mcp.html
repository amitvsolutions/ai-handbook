    Chapter 11 · The Universal Plug

    Model Context Protocol: one standard for tools and data

    Every agent Exponential AI built reached its tools and data through bespoke, one-off integrations — a new adapter for every model-and-system pair. MCP replaces that M×N sprawl with a single open protocol: a universal plug that lets any model talk to any tool or data source through one well-defined interface. This is the chapter that tames the mess agents created.

    01 · Recap & Introduction

    Where Exponential AI left off — and why MCP comes now

    Let's be honest about the ordering. Agents (Chapter 8), agentic systems (Chapter 9), and LangGraph orchestration (Chapter 10) all quietly assumed the same thing: that an agent can reach a tool or data source. Each of those connections was hand-built — a custom wrapper around the orders API here, a bespoke database client there, a different auth dance for every system. MCP appears after agents not because it logically follows them, but because the pain it solves only becomes unbearable once you have many agents and many tools. The mess had to exist before the standard made sense.

    And the mess is quadratic. With M AI applications and N tools/data sources, naïve integration means up to M×N custom connectors, each written, secured, and maintained separately. The Model Context Protocol (MCP) — an open standard introduced by Anthropic in late 2024 — collapses that to M+N: every tool exposes itself once as an MCP server, every app speaks MCP once as a client, and any client can use any server. It is, deliberately, "the USB-C of AI integrations."

    What you'll be able to do after this chapter

    Explain the M×N → M+N argument; describe MCP's client–server model and its three primitives (tools, resources, prompts); contrast MCP with bespoke REST integrations; and see the boundary MCP doesn't cross — agent-to-agent communication — which sets up Chapters 12–13.

    02 · Why this technology exists

    Why the world invented MCP

    MCP exists because integration was quietly becoming the dominant cost of building with LLMs. Every team re-solved the same problems: how to describe a tool to a model, how to pass arguments and get results back, how to expose a document or database read-only, how to handle auth and errors. Each solution was subtly different, none were portable, and a tool built for one framework couldn't be reused by another. The industry was rebuilding the same adapters over and over.

    History rhymes here. Before USB, every peripheral had its own connector; before ODBC, every application wrote a driver per database; before HTTP, networked apps invented bespoke protocols. In each case a shared standard turned an M×N explosion into an M+N ecosystem and unlocked an explosion of interoperability. MCP is that move for AI tool-and-data access: standardise the interface once, and a tool written by anyone becomes usable by any MCP-speaking application — including Exponential AI's agents, no matter which framework they happen to run on.

    There is a second, subtler reason MCP exists: separation of ownership. When the orders team, the CRM team, and the security team each publish their own MCP server, they own the contract for their system — its scopes, its audit logging, its rate limits — independently of whichever AI application happens to consume it. The bespoke world coupled every integration to a specific app's release cycle and a specific engineer's memory. MCP makes the integration a first-class, independently versioned product with a clear owner, so Exponential AI can evolve its agents and its systems on separate schedules without either side breaking the other.

    03 · Problem statement

    The problem, stated precisely

    Given many AI applications and many external tools/data sources, provide a single, model-agnostic, framework-agnostic protocol for (a) discovering what a tool offers, (b) invoking actions with typed arguments, (c) exposing read-only context (documents, rows, files), and (d) doing all this with consistent auth, transport, and error semantics — so an integration written once works everywhere.

    The subtlety is discovery and self-description. It isn't enough to call a function; a client must be able to ask a server "what can you do?" and receive a machine-readable catalogue the model can reason about at runtime. MCP therefore defines not just a calling convention but a way for servers to advertise their capabilities, so agents can adapt to new tools without code changes. That dynamic, negotiated interface is what separates a real protocol from yet another SDK.

    Three levels of understanding

    One idea, three depths

    Level 1 · Simple

    A universal power socket

    Instead of a different plug for every device, one standard socket fits them all. MCP is that socket for AI: any tool that "fits" can be used by any AI app, no custom wiring.

    Level 2 · Engineering

    Client–server over JSON-RPC

    An MCP client (in the AI app) connects to MCP servers that expose tools, resources, and prompts. Messages use JSON-RPC; the client discovers capabilities and invokes them uniformly.

    Level 3 · Architect

    An interoperability layer

    MCP decouples AI apps from integrations. Tools become reusable, independently governed services; the M×N connector matrix becomes an M+N ecosystem with consistent security boundaries.

    04 · Historical evolution

    From bespoke wrappers to an open protocol

    MCP is young, but it landed on a well-worn path: capability explodes, integration cost explodes with it, then a standard emerges to absorb the chaos.

    2023 — Function calling

    Models gain structured tool calls — but each tool is wired per app, per framework.

    Early 2024 — Plugin sprawl

    Every platform ships its own plugin/tool format; nothing is portable across ecosystems.

    Late 2024 — MCP announced

    Anthropic open-sources MCP: a client–server standard with tools, resources, and prompts.

    2025 — Ecosystem growth

    MCP servers appear for filesystems, databases, SaaS, and internal systems; multiple vendors adopt the client side.

    05 · Internal architecture

    Inside MCP: hosts, clients, servers, primitives

    MCP uses a client–server model. A host is the AI application (an IDE assistant, a chat app, one of Exponential AI's agents). Inside it runs one or more MCP clients, each maintaining a connection to an MCP server. A server is a small program that wraps a specific system — the orders database, a filesystem, a SaaS API — and exposes its capabilities through the protocol. Communication is JSON-RPC messages over a transport (local stdio for local servers, or HTTP-based streaming for remote ones).

    Servers expose three primitives. Tools are actions the model can invoke (functions with typed inputs/outputs — "issue_refund", "search_orders"). Resources are read-only data the client can fetch and feed as context (files, records, documents). Prompts are reusable templates a server offers to guide the model for common tasks. A client first discovers what a server offers, then uses these primitives uniformly — the same way regardless of what system sits behind them. That uniformity is the whole point: the agent's code doesn't change when the tool does.

    06 · Step-by-step execution

    An Exponential AI agent uses an MCP server

    1. Connect. The agent's MCP client opens a connection to the "orders" MCP server.
    2. Discover. Client asks for capabilities; server returns tools (search_orders, issue_refund) and resources (policy docs).
    3. Plan. The model, seeing the catalogue, decides to call search_orders.
    4. Invoke. Client sends a JSON-RPC tool call with typed args; server executes and returns a result.
    5. Ground. The agent fetches a policy resource and adds it to context.
    6. Act. After checks, it calls issue_refund — the same protocol, a different tool.

    A capability handshake, sketched

    // client → server
{"jsonrpc":"2.0","id":1,"method":"tools/list"}

// server → client
{"jsonrpc":"2.0","id":1,"result":{"tools":[
  {"name":"search_orders","description":"Find orders by customer or id",
   "inputSchema":{"type":"object","properties":{"query":{"type":"string"}}}},
  {"name":"issue_refund","description":"Refund an order (idempotent)",
   "inputSchema":{"type":"object","properties":{"order_id":{"type":"integer"}}}}
]}}
    07 · Analogies

    Four ways to picture MCP

    USB-C. Once every laptop and every peripheral speaks the same port, you stop buying a special cable for each device. MCP is the USB-C port for connecting models to tools and data.

    A standard integration bus. Exponential AI's systems each expose one MCP server; any current or future agent plugs into the bus instead of every team re-integrating every system.

    ODBC/JDBC for AI tools: a common driver interface so applications talk to many backends through one contract, instead of a bespoke client per backend.

    The Language Server Protocol (LSP) pattern: one protocol let every editor talk to every language's tooling. MCP does the same for models and external capabilities.

    11 & 12 · Advantages & disadvantages

    Strengths and costs

    Advantages

    • M+N, not M×N — integration cost grows linearly.
    • Reusable tools across apps and frameworks.
    • Runtime discovery — agents adapt to new tools without code changes.
    • Consistent security boundary per server.

    Disadvantages

    • Young standard — tooling and patterns still maturing.
    • Extra moving part — servers to run, monitor, secure.
    • New attack surface — a malicious/over-scoped server is dangerous (Ch 18).
    • Overkill for a single app with one or two tools.
    13 & 14 · Best practices & anti-patterns

    Using MCP well — and badly

    Best practices

    • Scope servers narrowly — least privilege per system.
    • Write precise tool schemas & descriptions (they drive the model).
    • Make write tools idempotent and audited.
    • Vet third-party servers before granting access.

    Anti-patterns

    • One god-server exposing everything with broad rights.
    • Trusting unknown servers with sensitive scopes.
    • Leaking secrets through resource exposure.
    • Reinventing MCP with a private, non-portable format.

    Questions you should be able to answer

    Tools (callable actions the model invokes), resources (read-only context the client fetches), and prompts (reusable templates the server offers). Tools act; resources inform; prompts guide.

    Because the integration pain it removes only becomes severe once you have many agents and tools. MCP is a response to sprawl that agents created, not a prerequisite for a single agent.

    Agent-to-agent communication. MCP connects a model to tools and data; how independent agents discover and talk to each other is the domain of ACP (Ch 12) and A2A (Ch 13).

    16 · Architect notes

    Think like an architect

    Trade-off analysis

    MCP trades a new operational component for linear, reusable integration. The architect treats each MCP server as a governed micro-boundary: least-privilege scope, audited writes, and a clear owner. The payoff compounds as the number of agents grows.

    Failure scenario

    Exponential AI installs a convenient third-party MCP server with broad read scope; it quietly exposes customer PII as resources to every connected agent. MCP's power to connect is also a new attack surface — vetting and least-privilege are non-negotiable (Ch 18).

    17 · Common mistakes

    Where teams go wrong

    • Building a single over-privileged server instead of scoped ones.
    • Vague tool schemas that mislead the model.
    • Skipping server vetting and trusting unknown code.
    • Using MCP where a single inline function would do.

    19 · Checklist

    Before shipping an MCP integration

    • Servers are scoped to least privilege.
    • Tools have precise schemas and descriptions.
    • Write tools are idempotent and audited.
    • Resources don't leak secrets or PII.
    • Third-party servers vetted and pinned.
    • Auth and transport configured securely.

    21 · Summary

    Summary

    MCP is the universal plug for AI: an open client–server protocol that lets any model reach any tool or data source through one interface. By having tools expose themselves once as servers (offering tools, resources, and prompts) and apps speak the protocol once as clients, it turns the M×N integration explosion into an M+N ecosystem with runtime discovery and consistent security boundaries. For Exponential AI it ends the era of bespoke, throwaway connectors — and it appears after agents precisely because that sprawl is what made a standard necessary.

    But notice the shape of everything MCP does: it connects one model to its tools and data. It says nothing about how two independent agents — perhaps built by different teams, or different companies — discover each other, describe their capabilities, and collaborate. As Exponential AI's agents multiply and start needing to delegate to each other, a different kind of standard is required.

    22 · What happens next

    Transition to Chapter 12

    MCP gave every agent a clean way to reach tools — but Exponential AI's agents increasingly need to reach each other: the returns agent asking the fraud agent for a risk score, or an internal agent calling a supplier's agent. What standard lets autonomous agents communicate as peers, rather than as apps calling tools? That is the emerging category of agent communication protocols — Chapter 12 — ACP, where we look at the intent, IBM's ACP, and why the wire formats are still converging.
