    Chapter 21 · Build & Ship

    AIOps: building and deploying AI software

    The lifecycle (Ch 20) produced a ship-ready candidate and a definition of "good enough." AIOps is the engineering discipline that turns that candidate into a live service — repeatably and reversibly — through CI/CD for models and prompts, versioning and registries, and deployment patterns like canary, blue-green, shadow, and rollback.

    01 · Recap & Introduction

    Where Exponential AI left off

    Chapter 20 gave Exponential AI a methodology: frame the problem, build a frozen evaluation set, experiment, and promote a candidate only when it clears the bar on the slices that matter. The lifecycle's exit gate produced a concrete artefact — a chosen model or prompt-plus-model system — and a threshold that defines success. What it deliberately did not do was ship it. A candidate that scores well in a notebook is not a product; between "the eval passed" and "customers are using it" lies a whole engineering discipline.

    AIOps is that discipline: the build-and-ship mechanics for AI systems. It answers how you package a model and its prompts, version everything that affects behaviour, gate releases on evaluation, and roll changes out to production in a way that limits risk and can be undone instantly. If the SDLC is the recipe and the taste test, AIOps is the commercial kitchen that produces the dish the same way every service and can pull a bad batch before it reaches the table.

    What this chapter owns — and what it hands off

    This chapter owns build & ship: CI/CD for models and prompts, environments, versioning and registries, packaging and serving, deployment patterns (canary/blue-green/shadow/rollback), and release governance. It does not cover the overall methodology — that is Chapter 20 · AI SDLC. And it stops at go-live: once traffic is flowing, observability, SLOs, incidents, drift, and on-call belong to Chapter 22 · AI-SRE. Deploying is getting it out safely; keeping it healthy is the next chapter.

    02 · Why this technology exists

    Why deploying AI needs its own mechanics

    Ordinary CI/CD versions and ships code. AI systems have three more things that change behaviour and must be shipped together: the model (weights or a pinned API version), the prompts (which are effectively source code for an LLM), and the data the behaviour was derived from. Change any one and the output distribution changes, so "what version is in production?" is a four-part answer, not one git SHA. Traditional pipelines that track only code silently lose the provenance that makes AI releases reproducible and auditable.

    AIOps also exists because the release gate is different. In classic software, the gate is a deterministic test suite: green means correct. In AI, correctness is statistical, so the gate is an evaluation against the SDLC's frozen set — a candidate is promoted only if it clears the metric threshold on the slices that matter. A build can compile perfectly and still be worse than what's live, so the pipeline must block on quality regressions, not just on compilation and unit tests.

    Finally, the mechanics exist because the blast radius of a bad AI release is large and often invisible at first. A regressed model doesn't crash; it quietly makes worse decisions — more wrong refunds, more mis-tagged product images — which may not show up until customers notice. That is why AIOps leans so heavily on progressive delivery (expose new versions to a sliver of traffic first) and instant rollback (return to the last known-good version in seconds). You cannot rely on catching every regression before release, so you design the release so a missed regression is cheap to reverse.

    03 · Problem statement

    The problem, stated precisely

    Given a ship-ready candidate (model + prompts + the data lineage behind it) and an evaluation threshold from the lifecycle, build a repeatable pipeline that packages it, records exactly which versions are in play, verifies it still clears the bar, and rolls it out to production such that any regression is contained to a small fraction of traffic and can be reversed instantly — with a clear governance trail of who released what and why.

    The organising principle is reversible, progressive delivery gated on evaluation. Never flip 100% of traffic to a new version on the strength of an offline score alone. Ship behind a gate, expose it gradually while watching the signals, and keep the previous version one switch away. The goal is not to guarantee every release is perfect — it is to guarantee no release can do lasting damage before it is caught and rolled back.

    Three levels of understanding

    One idea, three depths

    Level 1 · Simple

    A test kitchen with an undo button

    Serve the new dish to a few tables first. If they love it, serve everyone; if not, switch back to the old recipe instantly. Nobody gets a bad meal at scale.

    Level 2 · Engineering

    Versioned artefacts + eval gate + canary

    Register the model, prompt, and data version; run the eval in CI; deploy to a canary slice; promote on healthy signals; roll back automatically on regression.

    Level 3 · Architect

    Reproducible, reversible release system

    Design a registry and pipeline so any production behaviour is traceable to exact versions, every release is progressive and instantly reversible, and governance is automatic, not manual.

    04 · Historical evolution

    From code deploys to model deploys

    Deployment practice matured from manual releases toward the reversible, progressive delivery AI needs.

    Manual releases

    Hand-copied builds; slow, risky, hard to reverse.

    CI/CD for code

    Automated build/test/deploy pipelines make code releases fast and repeatable.

    MLOps & model registries

    Models and data become versioned artefacts with lineage; pipelines retrain and package them.

    LLMOps & prompt CI

    Prompts, tools, and API model versions are treated as shippable artefacts, gated by evaluation and rolled out progressively.

    05 · Internal architecture

    Inside an AIOps pipeline

    A production AIOps setup has a few cooperating parts. A registry stores versioned artefacts — model weights or a pinned model ID, the prompt set, the data snapshot, and the evaluation results that justified promotion — so any deployed behaviour is fully traceable. A CI pipeline triggers on a change to any artefact: it builds the package, runs unit and integration tests, and — critically — runs the evaluation against the frozen set, blocking promotion on a regression. Environments (dev → staging → prod) let changes be validated on progressively more realistic traffic.

    Packaging & serving wraps the artefact behind a stable inference interface (a container or a serving endpoint) so callers don't change when the version does. Deployment control implements progressive delivery: route a small canary slice or run a shadow copy in parallel, compare against the incumbent, and promote or abort. A rollback mechanism keeps the previous version warm and switchable in seconds. Wrapping all of it is release governance: approvals, an audit trail, and policy checks (e.g., no model ships without a passing eval and a named owner). Note the deliberate seam — this architecture emits metrics and health signals for the canary decision, but it does not operate those signals long-term; consuming them for SLOs, alerting, and incident response is AI-SRE's architecture (Ch 22).

    06 · Step-by-step execution

    Exponential AI ships a new refund-triage prompt

    1. Change. An engineer improves the refund-triage prompt and opens a pull request.
    2. CI + eval gate. The pipeline builds the package and runs the frozen eval; the new prompt beats the incumbent on precision and on the "rare category" slice, so the gate passes.
    3. Register. The prompt version, the pinned model ID, and the eval report are written to the registry with an owner and timestamp.
    4. Canary. Deployment control routes 5% of live traffic to the new version, previous version kept warm.
    5. Compare. The canary's decision quality and cost signals are compared against the incumbent on the same traffic.
    6. Promote or roll back. Healthy → ramp to 100%. Regressed → flip back to the previous version in seconds; no customer sees sustained harm.
    7. Hand off. Once at 100%, ongoing health is watched by AI-SRE (Ch 22), whose signals may later trigger the next iteration back in the SDLC.

    A release gate: promote only if the eval clears the bar

    def release(candidate, registry, traffic):
    report = run_eval(candidate, FROZEN_EVAL)         # gate on quality
    if report.precision < BAR or report.worst_slice < BAR - 0.05:
        return "blocked: eval regression"
    version = registry.publish(candidate, report)     # versioned + audited
    traffic.canary(version, pct=5, keep_warm=registry.current)
    if traffic.canary_healthy(version):               # signals compared live
        traffic.promote(version, pct=100)
        return f"shipped {version}"
    traffic.rollback()                                # instant, known-good
    return "rolled back"
    07 · Analogies

    Four ways to picture AIOps

    A factory production line with QA gates and a "stop the line" cord. Every unit is built the same way, checked before it leaves, and the line can be reverted instantly if a batch is bad.

    A phased product rollout to a few stores before national launch: limited exposure, measured results, and a plan to pull the product if early stores report problems.

    Feature flags and canary deploys — but the "feature" is a model or prompt, and the pass/fail gate is a statistical evaluation rather than a boolean test suite.

    An artefact registry with immutable versions and lineage — like a container registry, but each entry binds model + prompt + data + eval report so any release is reproducible.

    11 & 12 · Advantages & disadvantages

    Investing in AIOps: benefits and costs

    Advantages

    • Reproducible releases with full lineage.
    • Regressions caught at a gate, not by customers.
    • Contained blast radius via canary/shadow.
    • Instant rollback to known-good versions.

    Disadvantages / realities

    • Pipeline complexity — more moving parts to maintain.
    • Eval infra dependency — the gate is only as good as the eval.
    • Registry discipline required to avoid version sprawl.
    • Serving cost of running canaries/shadows in parallel.
    13 & 14 · Best practices & anti-patterns

    AIOps done well — and badly

    Best practices

    • Version model, prompt, data, and eval together; pin API model IDs.
    • Gate every release on evaluation, not just unit tests.
    • Deploy progressively; keep the previous version warm.
    • Make rollback a single, tested action.
    • Automate governance — owner, approval, audit on every release.

    Anti-patterns

    • Shipping on an offline score straight to 100% traffic.
    • Unpinned model versions — behaviour changes under you.
    • Prompts edited in production with no version or review.
    • No rollback plan — "we'll fix forward" during an incident.

    Questions you should be able to answer

    Because behaviour depends on model, prompt, and data — not just code — so all four must be versioned and shipped together, and the release gate must be a statistical evaluation rather than a deterministic test suite.

    Canary routes a small % of live traffic to the new version; blue-green keeps two full environments and flips between them; shadow runs the new version in parallel with no user impact to validate on real traffic. Canary and shadow are the AI workhorses.

    Because AI regressions are silent — worse decisions, not crashes — and you can't catch every one offline. Designing releases to be instantly reversible means a missed regression is cheap rather than catastrophic.

    AIOps owns getting a version safely to go-live and being able to reverse it. Once traffic is flowing steadily, observability, SLOs, alerting, incidents, and drift detection are AI-SRE's job (Ch 22).

    16 · Architect notes

    Think like an architect

    Trade-off analysis

    Optimise for reversibility over release speed. A slightly slower pipeline that guarantees instant, tested rollback beats a fast one that ships irreversible changes. For AI, where regressions are silent, the cost of a bad release is measured in customer trust — so spend on canary infrastructure and known-good warm standbys, not on shaving minutes off deploys.

    Failure scenario

    A team upgrades to a newer API model version without pinning or re-evaluating. Overnight the provider changes defaults, refund-triage precision drops, and there's no registered previous version to fall back to. Pinned versions plus an eval gate plus a warm rollback would have made this a non-event.

    17 · Common mistakes

    Where teams go wrong

    • Treating prompts as config instead of versioned, reviewed artefacts.
    • Skipping the canary because the offline eval "looked fine."
    • No lineage — can't say which data/model produced a decision.
    • Blurring the boundary — building monitoring/alerting here instead of shipping mechanics (that's Ch 22).

    19 · Checklist

    Before a release reaches production traffic

    • Model, prompt, data, and eval report versioned in the registry.
    • API model IDs pinned; no floating "latest".
    • Evaluation gate passed on aggregate and key slices.
    • Progressive rollout (canary/shadow) configured.
    • Previous version warm; rollback is one tested action.
    • Release has a named owner and an audit trail.

    21 · Summary

    Summary

    AIOps is the build-and-ship act of the AI lifecycle. It takes the SDLC's chosen candidate and turns it into a live service through mechanics ordinary CI/CD doesn't cover: versioning the model, prompt, and data together in a registry with lineage; gating releases on evaluation rather than deterministic tests; and rolling out progressively — canary, blue-green, shadow — with instant, tested rollback. Because AI regressions are silent, the discipline optimises for reversibility and contained blast radius, not just release speed, all under automatic release governance.

    But shipping safely is only half of production. Once traffic flows, the system must be kept healthy: watched, measured against reliability targets, debugged when it fails in non-deterministic ways, and protected against silent drift. That operational discipline is the final act.
