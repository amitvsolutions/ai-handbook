    Chapter 13 · The Reference Peer Protocol

    A2A: Google's Agent-to-Agent protocol

    Chapter 12 described the category of agent communication. A2A makes it concrete. Google's Agent-to-Agent protocol gives peer agents a documented way to advertise themselves (Agent Cards), exchange long-running tasks, and stream progress — all while complementing, not replacing, MCP. This is the spec an architect at Exponential AI can actually read and build against today.

    ~18 min read Level 1 → Level 3 Mermaid + ASCII diagrams Quiz included
    01 · Recap & Introduction

    Where Exponential AI left off

    Chapter 12 made the case that agents need to talk as peers, and that "ACP" names an emerging category — discovery, capability profiles, async tasks, identity — whose wire formats are still converging. We deliberately refused to crown a single standard. But an architect can't build against a category; they need a documented specification with concrete message shapes. A2A (Agent-to-Agent), introduced by Google in 2025 and since moved toward open governance, is the most fully realised of these efforts, so we use it as our reference.

    A2A takes the abstract building blocks of Chapter 12 and pins them down: agents publish an Agent Card describing who they are and what they can do; a client agent sends a task to a remote agent; the remote agent works asynchronously and streams updates and artifacts back. Crucially, A2A is designed to sit alongside MCP: MCP connects an agent to its tools, while A2A connects that agent to other agents. Same enterprise, two complementary layers.

    Emerging-terminology note

    A2A is well-documented but still young and evolving. We treat it as the reference specification for peer agent communication and frame it against MCP (tools) and the broader ACP category (Ch 12). Specific field names may change as the spec matures — the concepts won't.

    02 · Why this technology exists

    Why A2A was created

    A2A exists to make the ACP vision buildable now, with backing and documentation, so enterprises stop inventing pairwise glue between agents. Google's framing is explicit: as organisations deploy many specialised agents — often from different vendors and frameworks — they need an open, vendor-neutral way for those agents to collaborate without exposing their internal memory, tools, or reasoning. A2A is designed around that principle of opaque agents: a peer interacts with what an agent offers, not with how it works inside.

    It also exists to reuse the boring, proven parts of the web instead of reinventing them. A2A builds on familiar foundations — HTTP, JSON-RPC 2.0, and Server-Sent Events for streaming — so it slots into existing enterprise infrastructure, auth, and observability. That pragmatism is a big reason it became the reference: it feels like normal web engineering with agent-shaped semantics layered on top, which is exactly what a company like Exponential AI can adopt without a green-field rebuild.

    03 · Problem statement

    The problem, stated precisely

    Provide an open, transport-friendly protocol by which one agent can discover another, understand its capabilities, delegate a possibly long-running task, receive streamed progress and results, and do so across framework and organisational boundaries — without either agent exposing its internal implementation.

    The design tension A2A must resolve is autonomy vs interoperability. Agents should remain opaque and independently built, yet still collaborate reliably. A2A's answer is a small, well-defined surface — Agent Card for discovery, a task lifecycle for work, messages and artifacts for content — that says everything two agents must agree on and nothing about what happens inside either one. That minimal contract is what lets a Google-framework agent and a Python agent and a supplier's agent all interoperate.

    Three levels of understanding

    One idea, three depths

    Level 1 · Simple

    A business card + a work ticket

    Each agent has a "business card" (what it does, how to reach it). To get help you hand a peer a "ticket," and it works on it and reports back — even if it takes a while.

    Level 2 · Engineering

    Agent Card + task lifecycle over HTTP

    Agents publish a JSON Agent Card; clients send tasks via JSON-RPC over HTTP and receive streamed updates via SSE. Tasks carry state (submitted → working → completed) and artifacts.

    Level 3 · Architect

    Opaque peers, complementary to MCP

    A2A standardises peer collaboration between opaque agents while MCP standardises tool access. Together they form the "agent uses tools + agents work together" stack.

    04 · Historical evolution

    From proprietary glue to an open reference

    A2A is the point where the peer-communication category got a serious, documented anchor.

    2023–24 — In-framework only

    Multi-agent systems collaborate, but only inside one framework; cross-vendor is bespoke.

    Late 2024 — MCP

    Tool access standardised; agent↔agent left open on purpose.

    2025 — A2A launched

    Google publishes A2A with Agent Cards, tasks, and streaming; broad partner support.

    2025+ — Open governance

    A2A moves toward vendor-neutral stewardship; positioned as complementary to MCP.

    05 · Internal architecture

    Inside A2A: cards, tasks, messages, artifacts

    A2A has a few core objects. The Agent Card is a JSON document (conventionally discoverable at a well-known URL) that advertises an agent's identity, capabilities/skills, endpoint, and auth requirements — this is how a client discovers and decides to use a peer. The interaction unit is a Task: a client creates one, and it moves through a lifecycle (submitted → working → input-required → completed / failed / canceled). Within a task, the parties exchange Messages (turns of the conversation), and the working agent produces Artifacts (the concrete outputs — a document, a structured result, a file).

    The roles are a client agent (initiates) and a remote/server agent (fulfils); either agent can play either role in different exchanges, which is what makes it symmetric. Transport is deliberately ordinary: JSON-RPC 2.0 over HTTP(S) for requests, and Server-Sent Events to stream task updates and partial artifacts for long-running work. Because remote agents are opaque, none of this exposes their tools, prompts, or memory — a client sees capabilities and results, not internals. And note the clean division of labour with Chapter 11: inside its own boundary, a remote agent may use MCP to call tools; across the boundary, it speaks A2A to peers.

    One design choice deserves emphasis because it drives everything else: A2A is message- and task-centric rather than function-centric. MCP thinks in terms of typed function calls with fixed input/output schemas — appropriate when the callee is a deterministic tool. A2A instead exchanges messages made of parts (text, structured data, files) within a task, because a peer agent may respond with prose, a partial result, a question, or a document, and may change what it returns as the conversation develops. This is why the protocol leans on streaming and a stateful lifecycle instead of a single typed return value: it is modelling a collaboration, not a subroutine. For Exponential AI, that distinction is practical — a supplier's pricing agent might negotiate, revise, and attach a contract, none of which fits neatly into one function signature.

    06 · Step-by-step execution

    Exponential AI delegates to a supplier's agent

    1. Discover. Exponential AI's procurement agent fetches the supplier agent's Agent Card.
    2. Check skills. The card lists a "restock_quote" skill with its input/output schema and auth.
    3. Create task. The client sends a JSON-RPC request opening a task with the SKU and quantity.
    4. Stream. The supplier agent returns working and streams updates via SSE.
    5. Clarify. It moves to input-required ("need delivery date"); the client replies with a message.
    6. Deliver. The task reaches completed with an artifact: a structured quote.

    A2A-style task call (illustrative)

    // client agent → remote agent (JSON-RPC 2.0 over HTTP)
{"jsonrpc":"2.0","id":"7","method":"tasks/send",
 "params":{"id":"task-42","message":{"role":"user",
   "parts":[{"type":"text","text":"Quote 500 units of SKU-19 for restock"}]}}}

// remote agent streams updates (SSE), then final artifact
{"status":"working"}
{"status":"input-required","message":"Delivery date?"}
{"status":"completed","artifact":{"unit_price":4.20,"lead_days":9}}
    07 · Analogies

    Four ways to picture A2A

    Hiring a contractor: you read their profile (Agent Card), send a job (task), get progress calls (streaming), answer questions (input-required), and receive deliverables (artifacts) — without watching how they work.

    A purchase-order workflow between companies: a standard document format lets any buyer's system talk to any supplier's system without a custom integration each time.

    An OpenAPI/Swagger document for agents: the Agent Card is the discoverable contract, and JSON-RPC over HTTP is the familiar call mechanism.

    Microservices with service discovery: agents register cards, discover peers, and call each other with async, streamed responses — but the callees are reasoning agents, not CRUD services.

    11 & 12 · Advantages & disadvantages

    Strengths and costs

    Advantages

    • Documented & buildable now, with partner support.
    • Web-native — HTTP, JSON-RPC, SSE fit existing infra.
    • Opaque agents preserve autonomy and IP.
    • Complements MCP cleanly (tools vs peers).

    Disadvantages

    • Still evolving — fields and details may change.
    • Not yet universal — not every vendor speaks it.
    • Cross-org trust/security remain hard.
    • Operational weight for simple in-process cases.
    13 & 14 · Best practices & anti-patterns

    Using A2A well — and badly

    Best practices

    • Keep agents opaque; expose skills, not internals.
    • Use MCP inside, A2A across the boundary.
    • Model long work as tasks with streamed updates.
    • Secure Agent Cards & endpoints; authenticate peers.

    Anti-patterns

    • Leaking internal tools through the peer interface.
    • Using A2A for plain tool calls (that's MCP).
    • Blocking on long tasks instead of streaming.
    • Trusting unverified Agent Cards.

    16 · Architect notes

    Think like an architect

    Trade-off analysis

    A2A is the pragmatic bet: documented, web-native, complementary to MCP. Adopt it as your reference peer protocol, but keep it behind the same adapter you built in Ch 12 so the broader ACP category can still evolve underneath you. Use MCP inside each agent, A2A between agents.

    Failure scenario

    Exponential AI exposes an internal MCP tool directly to an external partner "to save time." The partner now has a raw handle into a backend system with no opaque boundary and weak audit. The fix: partners talk A2A to an opaque agent; that agent uses MCP internally under least privilege (Ch 18).

    17 · Common mistakes

    Where teams go wrong

    • Conflating A2A with MCP — using one where the other fits.
    • Breaking opacity by exposing internal tools to peers.
    • Ignoring the task lifecycle and blocking on long work.
    • Trusting Agent Cards without verifying identity.

    21 · Summary

    Summary

    A2A turns the ACP category into something an architect can build against. Agents publish an Agent Card for discovery, exchange work as long-running tasks with a clear lifecycle, stream updates over SSE, and return artifacts — all while staying opaque, so autonomy and IP are preserved. It reuses web-native transports (HTTP, JSON-RPC, SSE) and, crucially, complements MCP rather than competing with it: MCP inside an agent for tools, A2A between agents for collaboration. For Exponential AI, that's the full connectivity stack — an agent that can both use capabilities and work with other agents across boundaries.

    And yet connectivity is not competence. MCP, ACP, and A2A make it possible for a model to reach tools, data, and peers — but they say nothing about what information you should actually place in front of the model on any given turn. A returns agent with perfect protocols and a badly assembled context window still fails. Deciding what goes into that finite window, and in what shape, is its own discipline.

    22 · What happens next

    Transition to Chapter 14

    Here is the honest ordering bridge. Chapters 11–13 gave Exponential AI the plumbing: access to tools (MCP) and communication with peers (ACP/A2A). But plumbing only delivers information — it doesn't decide what belongs in the model's limited attention. Context engineering is the deliberate discipline of assembling exactly the right context — instructions, retrieved knowledge, tool results, memory — into that finite window.
