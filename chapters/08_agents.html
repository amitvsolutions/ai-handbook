    Chapter 08 · The Model Grows Hands

    AI Agents: from answering questions to taking action

    A RAG system can tell a customer their refund is eligible; an agent can actually process it. By giving an LLM tools and a loop — think, act, observe, repeat — Exponential AI turns a read-only oracle into a system that queries records, calls APIs, and completes tasks end-to-end. This is where AI stops talking and starts doing.

    ~18 min read Level 1 → Level 3 Mermaid + ASCII diagrams Quiz included
    01 · Recap & Introduction

    Where Exponential AI left off

    Chapter 7 connected Exponential AI's LLM to its knowledge: the assistant could now find the right fact and answer faithfully, with citations. But it remained a read-only oracle. A customer asking "refund my damaged order" got a correct explanation of whether they qualified — and then a human still had to open three systems and actually do it. Knowing is not doing.

    An agent closes that gap. Give the LLM a set of tools (functions it can call — query the order DB, issue a refund, send an email) and wrap it in a loop where it reasons about what to do, calls a tool, observes the result, and decides the next step until the goal is met. The model becomes the controller of a small program it writes on the fly. This chapter introduces the single-agent tool-use pattern; the next scales it into planning and autonomy.

    What you'll be able to do after this chapter

    Explain tool/function calling and the reason–act (ReAct) loop; describe an agent's core components (LLM, tools, memory, control loop); reason about when an agent is warranted vs a fixed workflow; and see why multi-step autonomy demands the planning and orchestration of Chapters 9–10.

    02 · Why this technology exists

    Why the world invented agents

    Agents exist because real tasks are multi-step and stateful, and their steps depend on intermediate results you can't know in advance. "Resolve this support ticket" might require looking up the order, checking the policy, verifying stock for a replacement, issuing a refund, and emailing the customer — and which steps apply depends on what each lookup returns. A single prompt can't express that; you need a loop that decides as it goes.

    The enabling breakthrough is tool calling: modern LLMs can be given a catalogue of functions with typed arguments and will emit a structured request to call one when appropriate. That single capability transforms the model from a text generator into a reasoning core that can reach out, affect the world, read the consequences, and continue. Everything else in an agent is scaffolding around that loop.

    03 · Problem statement

    The problem, stated precisely

    Given a high-level goal, no fixed script, and a set of tools, produce a correct sequence of actions that achieves the goal — deciding at each step what to do next based on observations, and knowing when to stop. This is fundamentally different from RAG's single-shot "retrieve then answer"; it is a closed-loop control problem where the LLM is the policy.

    The hard parts are reliability and safety. The loop can go wrong in ways a one-shot answer cannot: it can loop forever, call the wrong tool, misread an observation, or take an irreversible action (a real refund) based on a hallucinated premise. So an agent is not just an LLM plus tools — it is an LLM plus tools plus guardrails, limits, and often a human checkpoint, themes that recur through Chapters 15 and 18.

    There is also a compounding-error problem unique to loops. If a single LLM step is right 95% of the time, a one-shot answer is quite reliable — but an eight-step agent multiplies those odds, and 0.95 to the eighth power is only about 66%. Errors accumulate, and worse, a mistake early in the loop poisons every observation that follows. This is why mature agent design leans so heavily on verification at each step, on tools that fail loudly rather than silently returning nonsense, and on keeping the number of autonomous steps as small as the task allows. Reliability is not a feature you add at the end; it is the constraint that shapes the whole architecture.

    Three levels of understanding

    One idea, three depths

    Level 1 · Simple

    An assistant that can use apps

    Instead of only chatting, the AI can click the buttons: look up an order, issue a refund, send an email — doing the task, not just describing it.

    Level 2 · Engineering

    LLM + tools + a reason–act loop

    The model receives tool schemas, decides which to call with what arguments, gets the result back as an observation, and iterates. The loop ends on success, failure, or a step limit.

    Level 3 · Architect

    An LLM as a control policy

    The agent is a closed-loop controller with a stochastic policy. Design for bounded steps, idempotent/reversible tools, observability, and human approval on irreversible actions.

    04 · Historical evolution

    From chatbots to tool-using agents

    The agent idea is old in AI, but it only became practical once LLMs could reliably follow instructions and emit structured tool calls.

    Pre-2022 — Scripted bots

    Rule-based chatbots follow fixed decision trees; any unforeseen path breaks them.

    2022 — ReAct

    Interleaving reasoning and actions lets an LLM plan and use tools in a loop — the template for modern agents.

    2023 — Function calling

    Model providers add native, structured tool calling, making agents robust enough for production.

    2023+ — Agent frameworks

    Libraries standardise tools, memory, and loops; the door opens to multi-step and multi-agent systems (Ch 9–10).

    05 · Internal architecture

    Inside an agent

    An agent has four parts. The LLM is the reasoning core that decides what to do. Tools are typed functions it may call — each with a name, description, and argument schema — that connect it to the world (databases, APIs, calculators, other models). Memory holds the running transcript of thoughts, actions, and observations so the model keeps context across steps, plus optionally longer-term state. The control loop orchestrates the cycle and enforces limits.

    The dominant pattern is ReAct: the model produces a Thought (reasoning), then an Action (a tool call), the system executes it and returns an Observation, and the cycle repeats until the model emits a final answer. The tool descriptions are part of the prompt, so writing them well — clear names, precise argument docs — is as important as any code. Poor tool descriptions are one of the most common reasons agents misbehave.

    Memory deserves special attention because it is where the finite context window (Chapter 5) bites hardest. Every thought, action, and observation is appended to the transcript, so a long task steadily fills the window — and a full window means rising cost, rising latency, and eventually forgotten early context. Practical agents therefore manage memory rather than just accumulating it: summarising older steps, dropping stale observations, and keeping only what the next decision needs. That discipline — deciding what belongs in the model's working memory at each moment — is important enough that Exponential AI will eventually treat it as its own engineering practice, the subject of Chapter 14.

    06 · Step-by-step execution

    An agent resolves a refund ticket

    1. Goal in. "Customer says order #48122 arrived damaged; resolve it."
    2. Thought → Action. "I need the order" → call get_order(48122).
    3. Observation. Delivered 12 days ago, marked damaged.
    4. Thought → Action. "Check policy" → call get_policy("damaged").
    5. Observation. Full refund within 30 days — eligible.
    6. Action (guarded). Call issue_refund(48122) — requires approval if > threshold.
    7. Finish. Email the customer and return a summary. Loop ends.

    The ReAct loop, as pseudocode

    def agent(goal, tools, max_steps=8):
    memory = [system_prompt(tools), user(goal)]
    for _ in range(max_steps):
        step = llm(memory)                 # Thought + (Action | Final)
        if step.is_final:
            return step.answer
        result = call_tool(step.action)    # execute the chosen tool
        memory.append(step)
        memory.append(observation(result)) # feed result back in
    return "stopped: step limit reached"   # safety bound
    07 · Analogies

    Four ways to picture an agent

    A capable intern with access to the company's systems and a task list. They don't need every step spelled out — they figure out the next action from what they just found, and ask for sign-off on big moves.

    Exponential AI's tier-1 support rep, automated: it reads the ticket, checks the systems, applies policy, and completes the resolution — escalating to a human only on exceptions.

    A REPL driven by the LLM: it writes a call, sees the output, and writes the next call. The difference from a script is that the control flow is generated at runtime, not written in advance.

    A feedback control loop where the LLM is the controller and tools are actuators and sensors. Observations close the loop; step limits and guards keep it stable.

    11 & 12 · Advantages & disadvantages

    Strengths and costs

    Advantages

    • Completes tasks, not just answers questions.
    • Handles variable paths without a hard-coded script.
    • Composes tools — DB, APIs, retrieval, code.
    • Extensible — add a capability by adding a tool.

    Disadvantages

    • Less predictable — stochastic control flow.
    • Can take wrong/irreversible actions without guards.
    • Costly & slower — many LLM calls per task.
    • Harder to debug & test than fixed code.
    13 & 14 · Best practices & anti-patterns

    Building agents well — and badly

    Best practices

    • Write excellent tool descriptions — they are prompt, not afterthought.
    • Bound the loop with step limits and timeouts.
    • Gate irreversible actions behind approval or dry-run.
    • Make tools idempotent and log every action for audit.

    Anti-patterns

    • Using an agent for a fixed process a workflow would do reliably.
    • Unbounded loops with no step or cost limit.
    • Giving write access with no human checkpoint.
    • Too many tools, overwhelming the model's choice.
    15 · Interview questions

    Questions you should be able to answer

    RAG retrieves and answers in one shot; an agent runs a loop, calling tools to take actions and using observations to decide the next step until the goal is complete.

    Thought (reason) → Action (tool call) → Observation (result), repeated. Interleaving reasoning with acting lets the model plan and adapt to intermediate results.

    Bound steps and cost, make tools idempotent, require approval for irreversible actions, validate tool inputs/outputs, and log everything for audit and rollback.

    When the process is stable and well-defined — a deterministic workflow is cheaper, faster, and more predictable than letting an LLM decide each step.

    16 · Architect notes

    Think like an architect

    Trade-off analysis

    Agents trade predictability for autonomy. The architect's rule of thumb: use the least autonomy that solves the problem. If the path is known, script it; reserve agentic loops for genuinely open-ended tasks, and always wrap them in limits and audit.

    Failure scenario

    An unguarded agent misreads an observation and issues duplicate refunds in a loop. Idempotent tools, a per-task action limit, and human approval above a dollar threshold would each have stopped it — which is why they are non-negotiable for write tools.

    17 · Common mistakes

    Where teams go wrong

    • Vague tool descriptions that lead the model to misuse tools.
    • No loop bounds, causing runaway cost or infinite loops.
    • Granting write access without approval gates or dry-runs.
    • Reaching for an agent where a simple workflow is safer and cheaper.

    19 · Checklist

    Before shipping an agent

    • Each tool has a clear name, description, and typed schema.
    • Loop has step, time, and cost limits.
    • Irreversible actions require approval or a dry-run.
    • Tools are idempotent; inputs/outputs validated.
    • Every action is logged for audit and rollback.
    • An agent is genuinely warranted over a fixed workflow.

    21 · Summary

    Summary

    An agent is an LLM given tools and a loop: it reasons, acts, observes, and repeats until a goal is met. Tool calling is the enabling breakthrough, and the ReAct pattern — Thought, Action, Observation — is its beating heart. For Exponential AI this turned a read-only assistant into a system that resolves tickets end-to-end. But autonomy is a cost: agents are less predictable and can take irreversible actions, so bounded loops, idempotent tools, audit logs, and human gates are essential, not optional.

    This chapter covered a single agent handling a bounded task. Real enterprise goals are bigger: plan a multi-stage project, coordinate specialised sub-tasks, recover from failures over long horizons. Scaling from one tool-using loop to genuine, planning-driven autonomy is the next step.
