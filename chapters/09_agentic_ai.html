    Chapter 09 · Planning & Autonomy

    Agentic AI: from one loop to systems that pursue goals

    A single agent resolves one ticket. Agentic AI tackles the whole operation — decomposing a goal into sub-tasks, planning, remembering across long horizons, coordinating specialised agents, and recovering from failure. This chapter is where Exponential AI moves from an automated task to an autonomous capability.

    ~18 min read Level 1 → Level 3 Mermaid + ASCII diagrams Quiz included
    01 · Recap & Introduction

    Where Exponential AI left off

    Chapter 8 gave Exponential AI an agent: an LLM with tools and a reason–act loop that resolves a single support ticket end-to-end. But the moment the goal grew — "run the entire returns-and-restocking process for the week" — the simple loop strained. That task has phases (triage, refund, restock, reorder, report), each with its own sub-steps, dependencies, and failure modes. One flat loop trying to hold all of it in a single transcript quickly loses the plot.

    Agentic AI is the step up: systems that exhibit goal-directed autonomy through explicit planning, persistent memory, and often multiple cooperating agents. Where Chapter 8's agent reacts step by step, agentic systems plan ahead, decompose work, delegate to specialists, and reflect on results to self-correct. The distinction is one of degree and structure — but at enterprise scale that structure is the difference between a demo and a dependable operation.

    What you'll be able to do after this chapter

    Distinguish an agent from an agentic system; explain planner–executor and multi-agent patterns; reason about single vs multi-agent trade-offs; recognise reflection and memory as reliability tools; and see why orchestrating all this reliably demands the framework of Chapter 10.

    02 · Why this technology exists

    Why agentic AI exists

    It exists because a flat reason–act loop degrades as tasks get longer and broader. Two forces push against it. First, the context window fills with the growing transcript, so a long task literally runs out of working memory. Second, the compounding-error problem from Chapter 8 worsens: the more steps, the lower the odds every one is correct. A single undifferentiated loop has no way to contain either problem.

    Structure is the answer. Planning turns a vague goal into an explicit list of sub-tasks, so the system knows where it is and what remains. Decomposition and delegation let each sub-task run in its own focused context — a "refund agent" needn't hold the "reorder agent's" details. Reflection lets the system check its own work and retry. Persistent memory carries state across steps and even sessions. Together these keep a long-horizon task tractable where a flat loop would collapse.

    It helps to see this as the same move software engineering has always made when a program grows too big to hold in one function: you factor it into modules with clear interfaces. A flat agent loop is one giant function whose "variables" all live in a single, ever-growing prompt; an agentic system is that logic refactored into cohesive units with explicit inputs, outputs, and a coordinator. The payoff is the same too — each part becomes independently testable, replaceable, and reasoned-about. Exponential AI can improve its refund agent without touching its reorder agent, and can point to exactly which unit failed when something goes wrong, instead of scrolling through one enormous transcript hunting for the moment it drifted.

    03 · Problem statement

    The problem, stated precisely

    Given a complex, open-ended goal, produce and execute a plan that decomposes it into manageable sub-tasks, allocates each to an appropriate capability, tracks progress and shared state, handles partial failures, and knows when the overall goal is achieved — all while keeping any single reasoning step within a manageable context and error budget.

    The central tension is autonomy vs control. More autonomy means the system handles more variety without human help, but also more ways to go wrong and less predictability. Agentic design is largely about buying autonomy safely: explicit plans you can inspect, bounded sub-tasks, reflection checkpoints, and clear hand-offs — so the system is powerful without being a black box that occasionally torches production.

    Three levels of understanding

    One idea, three depths

    Level 1 · Simple

    A team, not a solo worker

    Instead of one assistant doing everything, a manager breaks the job into pieces and hands each to a specialist, then checks the results — like a small project team.

    Level 2 · Engineering

    Plan, delegate, reflect, remember

    A planner produces sub-tasks; executor agents (often specialised, with their own tools) carry them out; a reflection step reviews output; shared memory tracks state across the whole run.

    Level 3 · Architect

    An orchestrated multi-agent system

    Design the topology (who talks to whom), the state model, failure/retry semantics, and termination. Autonomy is bounded by explicit plans, budgets, and checkpoints you can audit.

    04 · Historical evolution

    From single agents to agentic systems

    Agentic AI emerged rapidly once single-agent tool use worked and people pushed it at bigger goals — and hit the limits described above.

    2022–23 — Single ReAct agents

    One LLM, tools, one loop. Great for bounded tasks, brittle on long ones.

    2023 — Autonomous-agent experiments

    Projects like AutoGPT/BabyAGI add planning and task queues — ambitious, but unreliable without structure.

    2023 — Reflection & roles

    Self-critique and role specialisation (planner, worker, critic) markedly improve reliability.

    2024+ — Orchestration frameworks

    Graph-based orchestration (Ch 10) makes multi-agent workflows stateful, resumable, and debuggable.

    05 · Internal architecture

    The anatomy of an agentic system

    The dominant structure is planner–executor. A planner decomposes the goal into an ordered (or dependency-aware) list of sub-tasks. Executors — often specialised agents, each with a focused prompt and its own tools — carry out sub-tasks. A reflection or critic step evaluates outputs and can trigger a retry or re-plan. A shared state store / memory holds the plan, progress, and intermediate results so everything stays coordinated.

    Multi-agent topologies vary. A supervisor pattern has one orchestrator that routes work to workers and aggregates results — predictable and easy to audit. A network pattern lets agents talk peer-to-peer — flexible but harder to control. The right choice depends on how much structure the task tolerates. Whatever the topology, the recurring lesson is that explicit state and explicit hand-offs beat an ever-growing free-form transcript — which is precisely what motivates a real orchestration framework in Chapter 10.

    Memory in an agentic system is layered rather than monolithic. There is short-term working memory — what the currently active agent needs right now; shared task state — the plan, its progress, and the intermediate results every agent can read; and optionally long-term memory — durable knowledge that outlives a single run, such as "this supplier is chronically late." Keeping these separate matters: dumping everything into one bucket recreates the very context-window and confusion problems agentic design set out to solve. A well-designed system passes each agent the minimum it needs and no more — again foreshadowing the deliberate discipline of context engineering in Chapter 14.

    06 · Step-by-step execution

    Exponential AI runs weekly returns-and-restocking

    1. Plan. Planner decomposes: triage returns → issue refunds → restock → reorder low stock → report.
    2. Delegate. Each sub-task goes to a specialised executor with the right tools.
    3. Execute. The refund agent processes eligible returns; the inventory agent updates stock.
    4. Reflect. A critic checks totals and flags anomalies (e.g., a refund spike).
    5. Recover. Failed sub-tasks are retried or escalated; the plan updates.
    6. Aggregate. The supervisor compiles a report and marks the goal complete.

    Planner–executor, as pseudocode

    def agentic(goal):
    plan = planner(goal)                     # list of sub-tasks
    state = {}
    for task in plan:
        agent = route(task)                  # pick a specialist
        result = agent.run(task, state)      # each has its own tools
        if not critic(task, result):         # reflection / self-check
            result = agent.retry(task, state)
        state[task.id] = result              # shared memory
    return supervisor.summarise(state)
    07 · Analogies

    Four ways to picture agentic AI

    A project manager leading a small team of specialists. They plan the work, assign pieces, review deliverables, and re-assign when something fails — nobody holds the whole job in their head at once.

    Exponential AI's operations department: a manager (supervisor) coordinates returns, inventory, and procurement clerks (executor agents), each expert in one system, reporting up to a shared dashboard (state).

    A distributed system of microservices with an orchestrator: each service (agent) owns a bounded concern; a coordinator manages workflow, shared state, retries, and aggregation.

    A workflow engine where nodes are reasoning agents. The plan is the DAG; reflection is validation; memory is the shared context passed along edges.

    11 & 12 · Advantages & disadvantages

    Strengths and costs

    Advantages

    • Handles big, open-ended goals a single loop can't.
    • Focused contexts via decomposition and specialists.
    • Self-correction through reflection and re-planning.
    • Resilience — failures are contained and retried.

    Disadvantages

    • Much more complex to build and reason about.
    • Expensive & slow — many agents, many calls.
    • Coordination failures and error propagation.
    • Hard to debug without strong observability.
    13 & 14 · Best practices & anti-patterns

    Building agentic systems well — and badly

    Best practices

    • Make the plan explicit and inspectable.
    • Keep each agent focused with a narrow role and toolset.
    • Add reflection/critics on important outputs.
    • Budget everything — steps, cost, and time per run.

    Anti-patterns

    • Multi-agent for a simple task — needless complexity.
    • No shared state model — agents drift out of sync.
    • Unbounded autonomy without budgets or checkpoints.
    • Agents that all do everything instead of specialising.

    16 · Architect notes

    Think like an architect

    Trade-off analysis

    Agentic AI trades simplicity for reach. The architect's discipline: add structure only to buy back reliability you actually need. Explicit plans, focused agents, reflection, and budgets are how you get autonomy without an unaccountable black box.

    Failure scenario

    Peer agents pass a subtly wrong stock figure back and forth with no shared source of truth; the error amplifies until reorders are wildly off. A supervisor topology with a single authoritative state store would have caught it — a direct argument for real orchestration (Ch 10).

    17 · Common mistakes

    Where teams go wrong

    • Reaching for multi-agent when one focused agent suffices.
    • No explicit plan or state, so runs are impossible to follow.
    • Skipping reflection, letting early errors propagate.
    • No budgets, so autonomous runs balloon in cost and time.

    19 · Checklist

    Before shipping an agentic system

    • The plan is explicit and inspectable.
    • Each agent has a focused role and toolset.
    • A shared state model tracks progress and results.
    • Reflection/critics guard important outputs.
    • Step, cost, and time budgets are enforced.
    • Failures re-plan, re-route, or escalate cleanly.

    21 · Summary

    Summary

    Agentic AI scales the single agent into a goal-directed system: a planner decomposes the goal, specialised executors carry out sub-tasks, a critic reflects and triggers retries, and shared persistent state keeps everything coordinated. This structure is how Exponential AI runs an entire multi-phase operation rather than a single ticket, containing the context-window and compounding-error limits that break a flat loop. The price is complexity, cost, and coordination risk — so autonomy should always be bounded by explicit plans, focused roles, reflection, and budgets.

    But notice what every reliable example above quietly assumed: an authoritative shared state, clean hand-offs, retries, and the ability to pause, resume, and inspect a run. Hand-rolling all of that with ad-hoc code is exactly where teams drown. Agentic AI needs a real orchestration substrate.
