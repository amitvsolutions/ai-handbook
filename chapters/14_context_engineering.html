    Chapter 14 · Designing the Window

    Context Engineering: what goes in front of the model

    Protocols can deliver any tool result, document, or peer message into an agent — but the model only ever sees a finite context window. Context engineering is the deliberate discipline of deciding what to put in that window, in what shape, and in what order, on every single turn. It is where Exponential AI's plumbing turns into performance.

    ~18 min read Level 1 → Level 3 Mermaid + ASCII diagrams Quiz included
    01 · Recap & Introduction

    Where Exponential AI left off — the ordering bridge

    Chapters 11–13 built the connectivity stack: MCP for tools and data, ACP and A2A for talking to other agents. Notice what all three have in common — they make information reachable. None of them decides what actually belongs in the model's attention on a given turn. That's the honest reason context engineering appears after the protocols: only once you can fetch anything from anywhere does the real problem become obvious — you can now fetch far more than the model can use, and stuffing it all in makes things worse, not better.

    The model's context window is a hard, finite budget of tokens. Everything competes for it: the system prompt, instructions, retrieved documents, tool outputs, conversation history, and memory. Context engineering is the practice of curating that budget — selecting, compressing, ordering, and formatting exactly the right information so the model can reason well. If prompt engineering (Ch 6) was about wording the instruction, context engineering is about assembling the entire payload around it.

    What you'll be able to do after this chapter

    Treat the context window as a managed budget; assemble system + instructions + knowledge + tool results + memory deliberately; apply compression, selection, and ordering; avoid "lost in the middle"; and see where enterprise knowledge modeling (context ontology) fits.

    02 · Why this technology exists

    Why context engineering became a discipline

    It exists because more context is not better context. As Exponential AI's agents gained retrieval and tools, the naïve instinct was to pour everything in: the full policy manual, the entire order history, every tool result. But long, noisy contexts degrade reasoning, raise cost and latency (every token is paid for — Ch 16), and cause the model to miss the one fact that mattered. The bottleneck moved from "can we get the information?" to "can we present the right information clearly?"

    There's also a reliability angle. Agents run many turns, and each turn re-assembles context. Without discipline, history balloons, stale facts linger, retrieved chunks contradict each other, and behaviour drifts unpredictably. Context engineering exists to make the assembled input deliberate and repeatable — a designed artifact rather than an accidental pile of whatever the plumbing happened to deliver.

    03 · Problem statement

    The problem, stated precisely

    Given a finite context window and far more potentially-relevant information than fits, select, compress, order, and format the subset that maximises the model's chance of producing a correct, grounded response for this specific turn — repeatably, at acceptable cost and latency, across long multi-turn interactions.

    Two constraints make it hard. First, capacity: even large windows are finite, and filling them costs money and time and can dilute attention. Second, position sensitivity: models attend unevenly, often weighting the beginning and end of the context more than the middle (the "lost in the middle" effect). So it isn't only what you include but where — the same facts placed badly can be effectively invisible.

    Three levels of understanding

    One idea, three depths

    Level 1 · Simple

    Packing a small backpack

    You can't take everything on the trip. Context engineering is choosing exactly the few items you'll actually need — and packing them so the important things are on top.

    Level 2 · Engineering

    Assembling the payload

    Build the context from parts: system, instructions, few-shot, retrieved knowledge, tool results, memory. Select and compress each to fit the token budget; order for salience.

    Level 3 · Architect

    A managed context pipeline

    Context becomes a first-class, versioned pipeline: retrieval + ranking + summarisation + memory policy + formatting, tuned against evals and cost, feeding every model call.

    04 · Historical evolution

    From prompt wording to context systems

    The field grew as windows grew and as agents made context dynamic rather than static.

    Prompt era

    Focus on wording a single, mostly-static prompt (Ch 6).

    RAG era

    Retrieval injects dynamic knowledge (Ch 7) — now context is assembled, not written.

    Agent era

    Multi-turn tool use and memory make context change every step — curation becomes essential.

    Context engineering

    Named as its own discipline: selecting, compressing, ordering, and governing what fills the window.

    05 · Internal architecture

    Anatomy of an assembled context

    A well-engineered context is built from distinct layers, each with a job. The system layer sets role, rules, and tone. The instruction layer states the current task. Few-shot examples demonstrate format and edge cases. The knowledge layer holds retrieved, ranked, and compressed documents (from RAG, Ch 7). The tool-result layer carries recent observations from MCP tool calls. The memory layer supplies relevant long-term facts and a summarised conversation history. Finally a scratchpad may hold the agent's working notes or plan.

    The engineering is in the policies that fill each layer under a shared budget: how many retrieved chunks and after what reranking; how aggressively to summarise history; which memories to recall; how to order the layers for salience (typically stable instructions first, the freshest and most task-critical material near the end). This is also where context ontology lives — modeling enterprise knowledge (entities like order, customer, policy and their relationships) so retrieval and assembly pull coherent, correctly-scoped facts rather than random text. We treat ontology here conservatively: it's enterprise knowledge modeling applied to context selection, not a single prescriptive standard.

    A useful way to see the whole architecture is as a competition for scarce space, arbitrated by a budget. Every layer wants more room — retrieval would happily add ten more chunks, history would keep every turn, tool output would paste its full JSON — but the budget is fixed, so the assembler must rank layers against each other, not just within them. That arbitration is the real engineering: it is not enough to summarise history well if doing so frees space that then gets wasted on low-value documents. The best context pipelines treat the window like a portfolio, continuously reallocating tokens to whichever layer yields the most reasoning value for this specific turn, and they make that policy explicit and testable rather than leaving it to whatever order the plumbing happened to append things in.

    06 · Step-by-step execution

    Assembling context for a returns query

    1. Fix the budget. Reserve tokens for output; the rest is the input budget.
    2. Anchor. Add system rules + the current instruction (stable, near the top).
    3. Retrieve & rank. Pull top policy/order chunks via RAG; rerank; keep the best few.
    4. Compress. Summarise long history and verbose tool output to fit.
    5. Recall memory. Add only the customer facts relevant to this turn.
    6. Order & format. Place the most task-critical material last; label sections clearly.

    A budgeted context assembler, sketched

    def build_context(query, budget):
    parts = [system_rules(), instruction(query)]        # stable, top
    docs  = rerank(retrieve(query))[:MAX_DOCS]          # select
    hist  = summarize(history) if too_long(history) else history
    mem   = recall_relevant(query)                      # not everything
    parts += [format(mem), format(docs), format(hist)]  # salient last
    return trim_to(parts, budget)                       # enforce budget
    07 · Analogies

    Four ways to picture context engineering

    Briefing a busy expert: you don't dump every file on their desk. You hand them a one-page brief with exactly the relevant facts, most important first. That brief is engineered context.

    An executive summary: analysts distill mountains of data into the few decision-relevant points, structured so the executive can act without reading everything.

    A CPU cache with a tiny budget: you can't hold all of memory, so you decide what's hot, evict the stale, and keep the working set that maximises hits.

    Framing a photograph: the skill isn't capturing everything, it's choosing what's in the frame and what's cropped out so the subject is unmistakable.

    11 & 12 · Advantages & disadvantages

    Why it pays off — and where it bites

    Advantages

    • Better accuracy — the right facts, clearly placed.
    • Lower cost & latency — fewer wasted tokens (Ch 16).
    • More reliable agents over long runs.
    • Reproducible behaviour — context is designed, not accidental.

    Disadvantages

    • Extra pipeline to build, tune, and evaluate.
    • Summarisation loss can drop the crucial detail.
    • Model-specific tuning (windows, position effects) differs.
    • Harder to debug — behaviour depends on assembled input.
    13 & 14 · Best practices & anti-patterns

    Engineering context well — and badly

    Best practices

    • Budget explicitly; reserve output tokens first.
    • Rerank then trim retrieved knowledge.
    • Summarise history instead of letting it grow unbounded.
    • Put the most critical facts where the model attends.
    • Evaluate context changes against a test set (Ch 17).

    Anti-patterns

    • "Stuff it all in" because the window is large.
    • Unbounded history that grows every turn.
    • Dumping raw tool output without compression.
    • Ignoring position — burying key facts in the middle.
    15 · Interview questions

    Questions you should be able to answer

    Prompt engineering words the instruction; context engineering assembles the entire per-turn payload — system, instruction, retrieved knowledge, tool results, memory — under a finite token budget, ordered for salience.

    Models tend to attend more to the start and end of the context than the middle, so information buried mid-context can be effectively ignored. Placement, not just inclusion, matters.

    Bigger windows still cost more per token, add latency, and can dilute attention with noise. Curation usually beats brute-force stuffing even when the window is large.

    As enterprise knowledge modeling — defining entities and relationships so retrieval and assembly select coherent, correctly-scoped facts. Treat it as a modeling aid, not a rigid standard.

    16 · Architect notes

    Think like an architect

    Trade-off analysis

    Treat context as a versioned pipeline with an explicit token budget, tuned against evals (Ch 17) and cost (Ch 16). The architect's job is to make "what the model sees" a controlled, observable artifact — because at Exponential AI's scale, a 10% context improvement compounds across millions of calls.

    Failure scenario

    To "be safe," a team injects the entire 40-page returns policy into every call. Cost triples, latency spikes, and accuracy drops because the one relevant clause is buried on page 30. Curation — three ranked, trimmed chunks — beats the whole manual.

    17 · Common mistakes

    Where teams go wrong

    • Equating a big window with "no need to curate."
    • Letting history grow unbounded across turns.
    • Pasting raw, verbose tool output into context.
    • Never measuring the effect of context changes.

    19 · Checklist

    Before shipping a context pipeline

    • Explicit token budget, output reserved.
    • Retrieval reranked and trimmed to top-k.
    • History summarised, not unbounded.
    • Tool output compressed.
    • Critical facts placed for salience.
    • Context changes measured against evals.

    21 · Summary

    Summary

    Context engineering is the discipline that turns reachable information into effective reasoning. The context window is a finite, paid-for budget; more is not better. By assembling deliberate layers — system, instruction, ranked knowledge, compressed tool results, and relevant memory — and ordering them for salience, Exponential AI gets more accurate, cheaper, and more reliable agents. It appears after the protocols because only once you can fetch anything does choosing what to show the model become the decisive skill; and it extends prompt engineering from wording an instruction to designing the entire payload, optionally guided by enterprise knowledge modeling (context ontology).

    But there's a danger hiding in all this power. We can now reach any tool, talk to any agent, and assemble any context — which means a mistake, an injection, or a bad instruction can now do real things: issue refunds, email customers, change records. Capability without constraint is a liability. Before Exponential AI lets agents act at scale, it needs to bound what they're allowed to do.
